# Tile Explorer Web — 项目亮点与设计思考

> 本文梳理项目在产品、工程和架构层面的亮点，帮助读者快速理解项目的设计理念与技术深度。

---

## 一、产品理念：极致纯净的解谜体验

### 无广告、零注册、打开即玩

本项目的核心差异化是**完全免费、无广告、无内购**。玩家不需要下载 App、不需要注册账号、不需要看任何广告——在浏览器里打开链接即可开始游戏。这与市场上主流三消游戏（依靠插屏广告和激励视频盈利）形成鲜明对比。

道具不通过付费或看广告获取，而是每通一关奖励一个——玩家需要策略性地选择和使用道具。

### PWA：浏览器里的原生应用

游戏实现了完整的 **Progressive Web App** 支持：

- 可以"安装到主屏幕"，在手机上获得与原生 App 几乎一致的体验
- 支持完全**离线游玩**（排行榜需要联网）
- 自定义启动画面和应用图标

这意味着用户获取成本极低——无需上架应用商店、无需下载安装，一个链接就能触达用户。

## 二、排行榜：零成本免登录的在线排名

### 技术方案

使用 **Supabase 免费层**（PostgreSQL + REST API）实现在线排行榜。整个后端成本为 **0 元/月**。

关键设计：

| 设计点 | 方案 |
|--------|------|
| 用户识别 | 首次访问自动生成 UUID，存 localStorage，无需注册 |
| 排名维度 | 总星数优先，同星数按最少步数排序 |
| 数据安全 | PostgreSQL 行级安全（RLS）+ SECURITY DEFINER 函数，客户端只有 anon 权限 |
| 离线支持 | 成绩先存本地队列，网络恢复后自动提交 |

### 为什么不用 IP 地址

我们明确排除了 IP 识别方案，原因包括：NAT 共享导致同一 WiFi 下多设备无法区分；移动网络 IP 频繁变化；IP 属于个人数据，收集它需要隐私政策。

### Supabase 免费配额 vs 预估用量

| 资源 | 免费配额 | 预估用量 |
|------|----------|----------|
| 数据库 | 500 MB | < 10 MB |
| API 请求 | 无限 | 仅在通关时触发 |
| 月活用户 | 50,000 MAU | 远低于此 |

## 三、数据驱动架构：表格化配置

### 难度曲线可编辑

难度不是写死在代码里的公式，而是一张**可编辑的关键帧表**：

```javascript
// js/difficulty.js — 直接编辑这张表即可调整难度曲线
export const DIFFICULTY_KEYFRAMES = [
  { level:   1, patternTypes:  3, setsPerType: 1, layers: 2 }, //   9 瓦片
  { level:   5, patternTypes:  6, setsPerType: 1, layers: 3 }, //  18 瓦片
  { level:  10, patternTypes:  8, setsPerType: 2, layers: 3 }, //  48 瓦片
  { level:  50, patternTypes: 20, setsPerType: 3, layers: 6 }, // 180 瓦片
  { level: 300, patternTypes: 28, setsPerType: 6, layers: 8 }, // 504 瓦片
  // ...共 18 条关键帧
];
```

策划或运营可以**直接修改这个文件来调整难度**，不需要碰任何游戏逻辑代码。中间关卡自动从前一条关键帧取值，也可以切换为线性插值模式实现平滑过渡。

### 道具系统完全声明式

每个道具的所有属性都在配置表中声明，游戏逻辑读取配置驱动行为：

```javascript
export const POWERUPS = {
  shuffle:  { icon: '🔄', label: '洗牌',   unlock: 1,  cap: 9, breaks3Star: true  },
  undo:     { icon: '↩️', label: '撤销',   unlock: 1,  cap: 9, breaks3Star: true  },
  hint:     { icon: '💡', label: '提示',   unlock: 1,  cap: 9, breaks3Star: false },
  trashOut: { icon: '🗑️', label: '扔垃圾', unlock: 5,  cap: 5, breaks3Star: true  },
  bomb:     { icon: '💣', label: '炸弹',   unlock: 10, cap: 5, breaks3Star: true  },
  freeze:   { icon: '❄️', label: '冰冻',   unlock: 51, cap: 3, breaks3Star: true  },
};
```

新增一个道具只需在配置表中添加一条记录 + 编写对应的执行函数，不需要修改 UI 渲染、奖励选择、星级判定等任何外围逻辑。

### 主题系统同样数据驱动

6 个主题（缤纷果园、奇趣动物园、春日庭院、星际探险、环球美食、像素冒险）各有独立的 32 emoji 图案库、背景图和背景音乐。每 3 关自动轮换主题。新增主题只需在 `THEMES` 数组中添加一个对象。

## 四、数学可解性保证

### 推导式难度公式

总瓦片数不是独立生成的，而是由公式**推导得出**：

```
tileCount = patternTypes × setsPerType × 3
```

这个看似简单的关系从根源上保证：每种图案的瓦片数一定是 3 的倍数，不存在无法消除的"剩余瓦片"。这个约束在 v1 版本中是缺失的，当时出现过 `15 ÷ 4 = 3.75` 这种无法分配的组合。

### 金字塔布局 + 求解器双重验证

每个关卡布局的生成经过两道验证：

1. **金字塔约束**：上层瓦片严格是下层的子集，确保玩家可以从底层开始逐层解锁
2. **求解器正向验证**：一个回溯搜索算法（运行在 Web Worker 中，不阻塞界面）尝试通关该关卡，只有确认至少存在一条通关路径才会使用该布局

求解器还会记录最优步数，用于星级评分的基准。

### 确定性种子

关卡号就是随机种子。同一关卡在所有设备上生成完全相同的布局，保证排行榜公平性，也使得关卡可以缓存——下次玩同一关直接读取缓存，不需要重新生成和求解。

## 五、程序化音效合成

### 零文件的音效系统

游戏中的所有交互音效（点击、消除、连击、道具、警告等）都是通过 **Web Audio API 实时合成**的，不需要任何音频文件。这意味着：

- **零网络请求**——不增加首屏加载时间
- **动态参数**——连击时音效可以实时升调，这是预录音频做不到的
- **体积极小**——音效代码只有几 KB

### 音乐理论驱动的连击音效

连击消除的音效不是简单的"哔哔"声，而是一套**基于 C 大调音阶的和弦递进系统**：

| 连击级别 | 和弦 | 波形 |
|----------|------|------|
| 1 | C 大三和弦 | 三角波（温暖） |
| 3 | Em 小三和弦 | 三角波 |
| 5 | G 大三和弦 | 三角波 |
| 6 | Am 小三和弦 | 锯齿波（明亮） |
| 8 | C 高八度 | 锯齿波 |
| 10（闪电） | 雷鸣 + 电击（白噪声 + 锯齿波扫频） | — |

从第 6 级开始波形从三角波切换到锯齿波，音色逐渐变亮，给玩家"蓄力"的感觉。闪电触发时是独立合成的双层音效（低频雷鸣 + 高频电击），与连击升调无关。

### BGM Ducking

当音效播放时，背景音乐自动降低 6dB（音量减半），200ms 后平滑恢复。这是专业游戏音频中的标准做法，确保音效不被 BGM 遮盖。

## 六、连击系统的迭代演进

连击系统经历了 **4 个版本**的迭代，每一版都解决了前一版的问题：

| 行为 | v1 | v2 | v3 | v4（当前） |
|------|----|----|----|-----------|
| combo +1 条件 | 任何三消 | 同图案连续三消 | 连击链内 + 链计数 ≥ 3 | 同 v3 |
| 切换图案是否清零 combo | — | 是 | 是 | **否** |
| 超时是否清零 combo | 是 | 是 | 是 | **否（已废弃超时）** |
| 闪电触发 | combo ≥ 5 自动 | 同 v1 | 满 10 手动释放 | **满 10 自动 + 归零** |

最终方案的设计哲学：**combo 只增不减**。玩家不会因为"切错了图案"或"想了一会儿"而受到惩罚。combo 在满 10 格时自动释放闪电（消除一组最稀有的图案），然后归零开始新一轮——形成一个持续的正反馈循环。

## 七、工程亮点

### 瓦片透明度：韦伯-费希纳定律

被覆盖的瓦片不是简单地变半透明，而是使用**指数衰减**来计算透明度：

```
alpha = 0.85^depth，最低不低于 0.4
```

这基于心理学中的**韦伯-费希纳定律**——人类对亮度的感知是对数关系的。线性递减（如 1.0, 0.8, 0.6, 0.4）在深层会显得"突然消失"，而指数递减每一层"看起来"变化量相同。0.4 的下限确保最底层的瓦片仍然可见。

### 槽位智能插入

瓦片飞入槽位时不是简单追加到末尾，而是**智能聚类**——如果槽位中已有同图案瓦片，新瓦片会插入到它的旁边。这让玩家一眼就能看出"还差一个就凑齐三个了"，不需要在 7 格中搜索。

### 覆盖关系计算：空间哈希

版面上可能有数百个瓦片，需要实时判断哪些被覆盖。如果用暴力遍历，复杂度是 O(n²)。实现中使用了**空间哈希**：将瓦片按网格坐标分组，每个瓦片只检查周围 9 个格子——复杂度降为 O(n)。

### 粒子对象池

消除特效使用预分配的粒子对象池（120 个粒子），用完后回收再利用。这避免了运行时的垃圾回收（GC）导致的帧率抖动，保证 60fps 稳定运行。

### 存储系统的分级缓存

localStorage 中有两种缓存：布局缓存（可以从确定性种子重新生成）和求解器结果缓存（计算成本高）。当存储空间不足时，**只淘汰布局缓存，永不淘汰求解器缓存**——这是一个基于数据再生成本的分级淘汰策略。

### Service Worker 三级缓存

- **静态资源**（JS/CSS/HTML）：安装时预缓存
- **CDN 资源**（PixiJS）：缓存优先，网络回退
- **主题媒体**（背景图、BGM）：**Stale-While-Revalidate**——先返回缓存版本（即时显示），后台静默更新。这是最先进的缓存策略，在离线支持和内容新鲜度之间取得最优平衡。

### 竞态安全

主题切换涉及异步加载背景图和 BGM。快速连续切换关卡时，可能后发起的请求先返回。实现中使用**递增令牌**来防护——每次切换生成新令牌，加载完成后对比令牌，只有令牌匹配才应用结果，过期结果直接丢弃。

### 双 CDN 容灾

PixiJS 从 jsDelivr 加载，如果加载失败自动回退到 cdnjs。两个 CDN 都由 Service Worker 缓存，首次成功后离线可用。

## 八、安全设计

| 安全措施 | 说明 |
|----------|------|
| Supabase anon key | 仅使用公开的 anon key，不使用 service role key |
| 行级安全（RLS） | 数据库表启用 RLS，只允许公开读取，写入通过 SECURITY DEFINER 函数 |
| 输入校验 | 客户端 + 服务端双重正则校验（支持中文名，1-12 字符） |
| XSS 防护 | 排行榜名称在插入 DOM 前经过 `escapeHtml()` 处理 |
| 名称清洗 | `sanitizeName()` 清除 Supabase RPC 不接受的字符，处理旧格式兼容 |

## 十、技术栈一览

| 层级 | 选型 | 理由 |
|------|------|------|
| 渲染引擎 | PixiJS v7.3.2（CDN） | GPU 加速、粒子系统、层叠点击检测 |
| 游戏交互 | PixiJS 全权负责 | 瓦片、槽位、道具按钮、粒子特效统一渲染 |
| 覆盖面板 | DOM（HTML/CSS） | 排行榜、设置等需要表单输入的场景 |
| 音效 | Web Audio API 程序化合成 | 零延迟、动态参数、连击升调 |
| BGM | MP3 文件 | 按主题组织，旋律质感 |
| 持久化 | localStorage | 游戏进度、道具、设置 |
| 排行榜 | Supabase 免费层 | PostgreSQL + REST，零成本 |
| 部署 | GitHub Pages | 零成本、全球 CDN |
| 离线支持 | Service Worker | 三级缓存策略 |

**零构建**：PixiJS 通过 CDN 引入，自有代码用原生 ES Module，没有 Webpack、没有 Babel、没有 npm build。整个项目可以直接双击 `index.html` 运行。
