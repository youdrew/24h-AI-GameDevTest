# 用户识别

## 方法：客户端 UUID

**主 ID**：首次访问时生成 UUID v4 并存储在 localStorage。所有后续请求都携带此 ID。

**显示名称**：自动生成为 `Player#XXXX`（4 位随机十六进制）。玩家可自定义（1–12 个字符，仅允许字母数字、下划线、连字符和中文）。

**客户端校验**：在提交排行榜前，`leaderboard.js` 对 `display_name` 进行正则校验（`^[a-zA-Z0-9_\-一-鿿 ]{1,12}$`），拒绝非法输入。服务端 RPC 函数同步进行相同校验（双重保障）。

**UUID 丢失处理**：如果 localStorage 被清除，玩家将被视为新用户，排行榜历史记录不可恢复。这是已知限制，为保持架构简洁不做额外恢复机制。

## 为什么不用 IP 地址

| 问题       | 说明                                                       |
|------------|------------------------------------------------------------|
| NAT 共享   | 同一 WiFi 下多个设备共享 IP，无法区分                      |
| 动态 IP    | 移动网络 IP 频繁变化，同一玩家 = 不同的"用户"              |
| 隐私       | IP 是个人数据；收集它需要隐私政策                          |
| 服务端     | IP 仅在服务端可用；静态网站无法访问                        |

## 为什么不用浏览器指纹

浏览器指纹存在不可靠性（浏览器更新、设置变化、扩展干扰）和隐私顾虑，且回收流程复杂度高于收益。因此采用简洁的 UUID 方案，UUID 丢失视为新用户。
